# Артефакты управления разработкой (Project Management)

**Цель:** Декомпозировать масштабную архитектуру на атомарные, оцениваемые и тестируемые задачи для Jira/YouTrack. Используется формат User Story + Acceptance Criteria (в стиле BDD: Given/When/Then).

## 5.1. Декомпозиция на Эпики и User Stories

### 📦 Epic 1: Core Infrastructure & Data Foundation (Недели 1-2)
*Фокус: Развертывание базовых сервисов и схем данных.*

**Story 1.1: Инициализация схемы БД и миграций**
*   **Как** Backend-разработчик, **я хочу** иметь готовые DDL-скрипты и Alembic-миграции, **чтобы** все сервисы работали с единой, типизированной структурой данных.
*   **Acceptance Criteria (AC):**
    1. Given чистая БД PostgreSQL 16, When запускается скрипт `init.sql`, Then создаются таблицы `campaigns`, `users`, `call_logs`, `agent_audit_logs` с правильными индексами (GIN для JSONB).
    2. When выполняется попытка вставить запись с невалидным UUID, Then БД возвращает ошибку валидации.

**Story 1.2: Настройка локального окружения (Docker Compose)**
*   **Как** Разработчик, **я хочу** запускать всю зависимость одной командой, **чтобы** не тратить время на ручную настройку окружения.
*   **AC:**
    1. When выполняется `docker-compose -f docker-compose.dev.yml up -d`, Then поднимаются PostgreSQL, Redis (с AOF), Qdrant, MinIO и Mock vLLM.
    2. When Mock vLLM получает запрос, Then он отвечает с искусственной задержкой 300 мс, имитируя реальную модель.

### 📦 Epic 2: Real-time Voice Pipeline (Недели 3-5)
*Фокус: Низколатентная обработка голоса и интеграция с телефонией.*

**Story 2.1: Реализация Streaming Voice Agent с Barge-in**
*   **Как** Пользователь системы, **я хочу**, чтобы бот мгновенно замолкал, если я начинаю говорить, **чтобы** диалог ощущался естественным, а не как общение с автоответчиком.
*   **AC:**
    1. Given активный звонок и воспроизводящийся TTS-ответ бота, When клиент начинает говорить (детектируется Silero VAD), Then TTS прерывается в течение 300 мс.
    2. When VAD срабатывает, Then ASR начинает потоковую транскрибацию, и новый промпт отправляется в LLM.

**Story 2.2: Интеграция Emotion Detector (OpenSMILE)**
*   **Как** Система, **я хочу** анализировать акустические признаки параллельно с ASR, **чтобы** адаптировать тон ответа.
*   **AC:**
    1. When аудиопоток обрабатывается, Then OpenSMILE извлекает фичи каждые 500 мс.
    2. When классификатор определяет `ANGRY` с уверенностью > 0.8, Then в состояние LangGraph (`AgentSessionState`) добавляется флаг `emotion_alert`, который модифицирует системный промпт LLM для следующей реплики.

### 📦 Epic 3: ML Core & LangGraph Orchestration (Недели 5-9)
*Фокус: Интеллект, планирование и оптимизация кампаний.*

**Story 3.1: Интеграция Propensity Model (CatBoost)**
*   **Как** LangGraph Orchestrator, **я хочу** получать отсортированный список пользователей с приоритетом, **чтобы** звонить сначала тем, кто с наибольшей вероятностью ответит и конвертируется.
*   **AC:**
    1. Given список из 1000 `user_id`, When отправляется POST-запрос на `/api/v1/ml/score/batch`, Then CatBoost-сервис возвращает ответ < 50 мс.
    2. Then ответ содержит поля `p_answer`, `p_conversion` и рассчитанный `priority_score`, отсортированные по убыванию.

**Story 3.2: Реализация узла HUMAN_APPROVAL в LangGraph**
*   **Как** Супервайзер, **я хочу** утверждать сгенерированные скрипты перед запуском, **чтобы** исключить риск отправки некорректных или незаконных сообщений клиентам.
*   **AC:**
    1. When граф достигает узла `HUMAN_APPROVAL`, Then его выполнение приостанавливается (`interrupt`), а состояние сохраняется в Redis.
    2. Then в Telegram-бот супервайзера приходит сообщение с текстом скриптов и inline-кнопками "✅ Approve" / "❌ Reject".
    3. When нажата "✅ Approve", Then граф возобновляет работу, переходя в узел `DIALING`.

**Story 3.3: Реализация Thompson Sampling (Bandit Optimizer)**
*   **Как** Система, **я хочу** динамически перераспределять звонки между вариантами скриптов, **чтобы** максимизировать конверсию по ходу кампании.
*   **AC:**
    1. Given 3 варианта скрипта с равными начальными весами (alpha=1, beta=1), When поступает 50 успешных звонков на `script_A` и 0 на `script_B`, Then алгоритм увеличивает вероятность выбора `script_A` до > 80% для следующих звонков.

### 📦 Epic 4: Memory, Reflection & Integrations (Недели 9-12)
*Фокус: Долговременная память, самообучение и связь с бизнес-системами.*

**Story 4.1: Интеграция эпизодической памяти (Mem0)**
*   **Как** Голосовой агент, **я хочу** знать историю предыдущих взаимодействий клиента, **чтобы** персонализировать приветствие и не задавать одни и те же вопросы.
*   **AC:**
    1. When начинается звонок для `user_id=123`, Then агент делает запрос `mem0.search` и получает факт: "Клиент просил не звонить утром".
    2. Then этот факт подставляется в `<MEMORY_CONTEXT>` промпта LLM.
    3. When звонок завершен успешно, Then `Reflection Agent` вызывает `mem0.add` с новым фактом, извлеченным из диалога.

**Story 4.2: PII Masking Middleware (Presidio)**
*   **Как** Security Officer, **я хочу**, чтобы все данные, проходящие через LLM и сохраняемые в логи, были очищены от ПДн, **чтобы** соответствовать 152-ФЗ.
*   **AC:**
    1. Given текст "Мой номер 89001234567", When он проходит через middleware `PIISanitizer`, Then на выходе получается "Мой номер `[PHONE_NUMBER]`".
    2. When данные сохраняются в `agent_audit_logs`, Then поле `input_payload` содержит только замаскированную версию.

**Story 4.3: Настройка Observability (Prometheus + Grafana)**
*   **Как** SRE-инженер, **я хочу** видеть дашборд с ключевыми метриками в реальном времени, **чтобы** мгновенно реагировать на деградацию качества сервиса.
*   **AC:**
    1. When система работает, Then в Prometheus доступны метрики `voicegraph_pipeline_latency_ms` и `voicegraph_call_outcomes_total`.
    2. When p95 latency превышает 800 мс в течение 2 минут, Then в Telegram-канал SRE приходит алерт с текстом "High Voice Latency Detected".
