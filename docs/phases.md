# План реализации: VoiceGraph: Autonomous Predictive Outbound Agent

## ЭТАП 0: Подготовительный и Compliance (Недели 1–2)
*Цель: Обеспечить юридическую чистоту, качество данных и базовую инфраструктуру до начала разработки ML/AI компонентов.*

### Подзадача 0.1: Аудит и нормализация данных CRM
- **Инструменты:** Python, `pandas`, `DVC` (Data Version Control), `Great Expectations`.
- **Реализация:** Создание пайплайна валидации исторических данных за 12 месяцев.
  - Модель валидации `CampaignDataSchema` (Pydantic): проверка наличия `user_id`, `phone_hash`, `consent_to_call` (bool), `last_contact_date`, `ltv_segment`.
  - Автоматическая фильтрация записей без флага согласия (строгое соблюдение 38-ФЗ).
- **Действие:** Настройка DVC-репозитория для версионирования сырых и очищенных датасетов. Запуск Great Expectations для генерации отчета о качестве данных (Data Quality Report).

### Подзадача 0.2: Инфраструктура телефонии и WebRTC-мост
- **Инструменты:** Rust, `Active Call` (или кастомный SIP-клиент), Asterisk, Docker.
- **Реализация:** Развертывание микросервиса-шлюза, принимающего WebRTC-поток от LiveKit и транслирующего его в SIP-транк Asterisk.
  - Включение SIPREC для дуплексной записи разговора.
  - Настройка Answering Machine Detection (AMD) на уровне Asterisk для отсечения автоответчиков до передачи потока в AI.
- **Действие:** Нагрузочное тестирование шлюза на 100 параллельных каналов (мониторинг CPU/RAM, потерянных пакетов).

### Подзадача 0.3: Pre-processing и маскирование PII (152-ФЗ)
- **Инструменты:** `Microsoft Presidio` (on-premise), Regex.
- **Реализация:** Создание асинхронного сервиса `PIISanitizer`.
  - Конфигурация распознавателей для РФ: паспортные данные (серия/номер), ИНН, номера банковских карт (Luhn algorithm), номера телефонов.
- **Действие:** Интеграция `PIISanitizer` как middleware: любой текст, идущий от ASR к LLM или в логи, сначала проходит через санитайзер и заменяется на токены `[PHONE_REDACTED]`, `[PASSPORT_REDACTED]`.

## ЭТАП 1: Real-time Voice Pipeline (Недели 3–5)
*Цель: Создать сверхнизколатентный (<800 мс) голосовой конвейер с поддержкой прерываний (barge-in).*

### Подзадача 1.1: Развертывание и оптимизация vLLM
- **Инструменты:** `vLLM`, Kubernetes, NVIDIA H100 (4x80GB), Qwen2.5-72B (или YandexGPT через API-шлюз).
- **Реализация:** 
  - Настройка `tensor_parallel_size=4` для распределения модели.
  - Включение `enable_prefix_caching=True` для кэширования системных промптов и истории диалога (ускоряет TTFT в 2-3 раза).
  - Настройка `max_num_seqs` и `gpu_memory_utilization=0.90` для максимизации throughput.
- **Действие:** Деплой сервиса и бенчмарк через `vLLM benchmark_serving` для подтверждения latency < 300 мс на первый токен.

### Подзадача 1.2: Сборка LiveKit Voice Agent (Streaming)
- **Инструменты:** `livekit-agents` (Python), `Faster-Whisper` (или Cohere Transcribe), `Silero TTS`.
- **Реализация:** Создание асинхронного воркера `VoiceAgentWorker`.
  - **VAD (Voice Activity Detection):** Использование `silero-vad` для мгновенного обнаружения начала речи клиента и прерывания TTS (barge-in < 300 мс).
  - **Streaming ASR:** Аудио чанками (200-400 мс) отправляется в Faster-Whisper, результат потоково передается в LLM.
  - **Streaming LLM:** LLM генерирует ответ токенами.
  - **Streaming TTS:** Как только накапливается ~15-20 токенов (конец предложения), они отправляются в Silero TTS для синтеза и воспроизведения.
- **Действие:** Реализация буферизации и синхронизации потоков, чтобы избежать "заикания" или наложения голосов.

### Подзадача 1.3: Мультимодальный анализ эмоций (Parallel Pipeline)
- **Инструменты:** `OpenSMILE` (или `librosa` + кастомная легковесная CNN), `asyncio`.
- **Реализация:** Параллельно с ASR, аудиопоток подается в экстрактор фич (тон, энергия, темп, паузы).
  - Модель классифицирует эмоцию в реальном времени: `Enum[CALM, ANNOYED, CONFUSED, ANGRY]`.
- **Действие:** Передача метки эмоции в `AgentSessionState` (см. ниже) для динамической адаптации промпта LLM (например, при `ANGRY` включать эмпатичный тон и предлагать звонок оператора).

## ЭТАП 2: Propensity Modeling & MLOps (Недели 5–7)
*Цель: Создать и внедрить ML-модель для интеллектуального приоритизации обзвона.*

### Подзадача 2.1: Feature Engineering и обучение CatBoost
- **Инструменты:** `catboost`, `featuretools`, `scikit-learn`.
- **Реализация:** 
  - Генерация 50-80 признаков: `hours_since_last_call`, `prev_call_duration`, `ltv_quantile`, `is_weekend`, `historical_answer_rate_{7,30,90}d`.
  - Обучение двух таргетов: `p_answer` (вероятность взять трубку) и `p_conversion` (вероятность успешного NPS/продажи).
  - Итоговый скор: `priority_score = p_answer * p_conversion`.
- **Действие:** Валидация на отложенной выборке (Target: ROC-AUC > 0.85).

### Подзадача 2.2: Калибровка и FastAPI-сервис инференса
- **Инструменты:** `FastAPI`, `uvicorn`, `scikit-learn` (CalibratedClassifierCV).
- **Реализация:** 
  - Применение Platt Scaling к выходным вероятностям CatBoost, чтобы скор 0.8 реально означал 80% вероятность.
  - Упаковка модели в Docker. Эндпоинт `/predict` принимает батч до 1000 `user_id` и возвращает отсортированный список с приоритетами.
  - Гарантия latency инференса < 10 мс на запрос.
- **Действие:** Регистрация модели и метрик в **MLflow** с настройкой RBAC (Data Scientist может обновлять, DevOps только деплоить).

## ЭТАП 3: LangGraph Оркестрация и Бандиты (Недели 7–9)
*Цель: Реализовать детерминированный, управляемый граф принятия решений для всей кампании.*

### Подзадача 3.1: Управление состоянием кампании (Campaign State Management)
- **Инструменты:** `langgraph`, `pydantic`, `Redis` (AOF persistence).
- **Реализация:** Создание строго типизированной модели `CampaignState`:
  ```python
  class CampaignState(TypedDict):
      campaign_id: str
      target_goal: str # например, "NPS_Optimization"
      candidate_pool: List[Dict] # Отсортированный список {user_id, priority_score, best_time}
      active_scripts: List[ScriptVariant] # 3-5 вариантов сценариев
      bandit_weights: Dict[str, float] # Веса для Thompson Sampling
      human_approval_status: Enum[PENDING, APPROVED, REJECTED]
      reflection_insights: List[str] # Накопленные инсайты
  ```
- **Действие:** Реализация `CheckpointSaver` от LangGraph на базе Redis для сохранения состояния графа после каждого шага (устойчивость к перезапускам).

### Подзадача 3.2: Проектирование узлов графа (StateGraph Nodes)
- **Реализация графа:**
  1. **`planner_node`:** Принимает цель. Вызывает Propensity API для получения `candidate_pool`. Генерирует 3-5 вариантов скрипта через LLM (с разными opening lines).
  2. **`human_approval_node`:** Приостанавливает граф (`interrupt`). Отправляет задачу в Telegram-бот супервайзеру: "План на 10 000 звонков. Скрипты: [ссылка]. Утвердить?". Ждет `approve`/`reject`.
  3. **`dialer_node`:** Читает очередь Redis Streams. Для каждого звонка выбирает скрипт на основе `bandit_weights`. Запускает `VoiceAgentWorker` (Этап 1).
  4. **`optimizing_node`:** После каждых N звонков обновляет веса скриптов через алгоритм Томпсона (Thompson Sampling).
  5. **`reflecting_node`:** Анализирует транскрипты отказов, генерирует инсайты, обновляет `reflection_insights` в состоянии.
- **Действие:** Написание unit-тестов для каждого узла с моками LLM и внешних API.

### Подзадача 3.3: Реализация Thompson Sampling (Bandit Optimizer)
- **Инструменты:** `numpy`, `scipy.stats.beta`.
- **Реализация:** Для каждого `script_id` храним параметры `(alpha, beta)`, изначально `(1, 1)`.
  - При успехе: `alpha += 1`. При неудаче: `beta += 1`.
  - Выбор скрипта для следующего звонка: сэмплируем `theta ~ Beta(alpha, beta)` для каждого скрипта и выбираем скрипт с максимальным `theta`.
- **Действие:** Интеграция этого алгоритма в `optimizing_node` для динамического перераспределения трафика в реальном времени.

## ЭТАП 4: Интеграция эпизодической памяти (Mem0) (Недели 9–10)
*Цель: Наделить агента "долговременной памятью" для гиперперсонализации.*

### Подзадача 4.1: Настройка Mem0 и Qdrant
- **Инструменты:** `mem0ai`, `qdrant-client`, Docker.
- **Реализация:** Развертывание Qdrant в режиме on-premise. Инициализация клиента Mem0 с конфигурацией:
  - `vector_db`: Qdrant
  - `llm`: локальный Qwen2.5-7B (для извлечения фактов из текста, чтобы не светить ПДн во внешние LLM).
- **Действие:** Создание индекса в Qdrant с payload-схемой: `user_id`, `fact` (string), `timestamp`, `decay_factor` (float, уменьшается со временем).

### Подзадача 4.2: Auto-Recall и инъекция в контекст
- **Реализация:** В начале звонка `VoiceAgentWorker` вызывает `mem0.search(query="last interactions, preferences, complaints", user_id=client_id, limit=5)`.
  - Результаты сортируются с учетом `decay_factor` (недавние факты имеют больший вес).
  - Формируется блок `<MEMORY_CONTEXT>` и внедряется в системный промпт LLM.
- **Действие:** Реализация пост-обработки: после звонка `Reflection Agent` вызывает `mem0.add(fact="Клиент упомянул, что переехал", user_id=client_id)` для обновления базы.

## ЭТАП 5: Reflection Agent и самообучение (Недели 10–11)
*Цель: Автоматизировать анализ ошибок и улучшение системы без участия человека.*

### Подзадача 5.1: Сбор мета-данных и триггер рефлексии
- **Инструменты:** PostgreSQL, `asyncpg`, Celery (или Redis Streams consumer).
- **Реализация:** После завершения звонка асинхронно записывается лог: `user_id`, `script_id`, `duration`, `outcome` (SUCCESS/REFUSAL/HANGUP), `transcript` (PII-masked), `max_sentiment_score`.
- **Действие:** Если `outcome != SUCCESS` или `sentiment_score == ANGRY`, задача отправляется в очередь `reflection_queue`.

### Подзадача 5.2: LLM-анализ и кластеризация инсайтов
- **Инструменты:** YandexGPT / Qwen2.5-72B, Pydantic Structured Output.
- **Реализация:** `Reflection Agent` получает транскрипт и промпт: *"Проанализируй этот отказ. Был ли вопрос неуместным? Был ли тон агрессивным? Предложи 1 конкретное изменение в скрипте."*
  - Ответ парсится в строгую Pydantic-модель `ReflectionInsight` (root_cause, suggested_script_change, confidence).
- **Действие:** Еженедельный джоб агрегирует эти инсайты, кластеризует их (например, "30% отказов из-за упоминания доставки") и автоматически обновляет Few-Shot примеры в промпте `planner_node` для будущих кампаний.

## ЭТАП 6: Интеграции, Observability и сдача (Недели 11–12)
*Цель: Замкнуть контур с бизнес-системами и обеспечить полную прозрачность работы.*

### Подзадача 6.1: Интеграция через Composio (CRM & Tools)
- **Инструменты:** `composio-core`, Битрикс24 / amoCRM API.
- **Реализация:** 
  - Настройка OAuth-приложения в Composio.
  - Реализация инструментов для LangGraph: 
    - `composio_bx24_create_deal`: создание сделки при успешном NPS/продаже.
    - `composio_bx24_add_task`: создание задачи "Перезвонить" при статусе "Сложный вопрос" с приоритетом High.
- **Действие:** Реализация механизма Circuit Breaker: если CRM недоступна (5xx ошибки > 5 за минуту), агент временно сохраняет результаты в Redis и синхронизирует позже, не прерывая звонки.

### Подзадача 6.2: Observability и дашборды
- **Инструменты:** Prometheus, Grafana, OpenTelemetry (OTel).
- **Реализация:** 
  - Инструментация FastAPI и LiveKit-воркеров через OTel.
  - Экспорт метрик: `voice_latency_ms`, `asr_wer` (Word Error Rate), `bandit_script_weights`, `campaign_conversion_rate`.
- **Действие:** Создание Grafana-дашборда "VoiceGraph Real-time Ops" с алертами в Telegram (например, "Latency > 1000ms в течение 2 минут" или "Доля отказов по скрипту А превысила 80%").

### Подзадача 6.3: Генерация финального отчета
- **Инструменты:** `WeasyPrint` (или `ReportLab`), `python-telegram-bot`.
- **Реализация:** По триггеру завершения кампании `Reflection Agent` собирает данные из PostgreSQL и формирует PDF:
  - График конверсии по скриптам (результат работы Bandit).
  - ТОП-5 причин отказов (из кластеризации).
  - Рекомендации для следующей кампании.
- **Действие:** Автоматическая отправка PDF супервайзеру в Telegram и прикрепление его как файла к сделке/контакту в CRM.

### Критерии готовности к переходу между этапами (Definition of Done)
1. **Этап 0 → 1:** DVC-датасет готов, compliance-чеклист подписан, тестовый SIP-звонок проходит без потерь пакетов.
2. **Этап 1 → 2:** Latency голосового пайплина стабильно < 800 мс (p95) при 100 параллельных сессиях, barge-in работает корректно.
3. **Этап 2 → 3:** CatBoost-модель развернута, ROC-AUC > 0.85, API отвечает < 10 мс.
4. **Этап 3 → 4:** LangGraph успешно проходит сквозной тест от планирования до обзвона на синтетических данных, Bandit-алгоритм корректно смещает веса.
5. **Этап 4 → 5:** Агент успешно использует исторические факты из Mem0 в диалоге (проверено на тестовом наборе "слепой" vs "с памятью").
6. **Этап 5 → 6:** Reflection Agent генерирует валидные JSON-инсайты, которые успешно обновляют промпты.
7. **Финал:** CRM обновляется в реальном времени, дашборды отображают метрики, PDF-отчет приходит автоматически. Все PII замаскированы.
