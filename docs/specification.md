# Autonomous Predictive Outbound Agent (Предиктивный AI-агент для кампаний / NPS / Canvassing)

## 1. Описание конечной системы
Система представляет собой **полностью автономного цифрового сотрудника (AI Campaign Manager)**, закрывающего весь жизненный цикл исходящих коммуникаций: от формирования гипотезы и предиктивной сегментации базы до проведения полнодуплексного голосового диалога и генерации бизнес-инсайтов. В отличие от классических predictive dialers, система не использует статические скрипты. Она динамически генерирует сценарии под конкретного клиента, используя его эпизодическую память (Mem0), предсказывает оптимальное время контакта через ML-модель (CatBoost) и обучается на результатах каждого звонка через механизмы саморефлексии (Reflection) и многоруких бандитов (Thompson Sampling).

## 2. Бизнес-задача
Крупные контакт-центры и BPO сталкиваются с кризисом эффективности исходящих кампаний:
1. **Низкий Contact Rate:** До 70–75% звонков при «слепом» обзвоне не достигают цели.
2. **Жесткость скриптов:** Статические роботы не адаптируются под эмоциональное состояние или контекст клиента, что ведет к высокому Drop-off rate.
3. **Задержка инсайтов:** Аналитика по результатам кампаний собирается вручную и занимает дни или недели.
4. **Высокий CAC:** Стоимость привлечения или опроса одного клиента неоправданно высока из-за холостых затрат на телефонную связь и время операторов.

**Решение:** Внедрение автономного агента, который минимизирует холостые звонки за счет ML-прогнозирования и максимизирует конверсию за счет гиперперсонализированного GenAI-диалога в реальном времени.

## 3. Бизнес-метрики (KPIs)
*   **Contact Rate:** Рост с базовых 25% до **> 40%**.
*   **Completion Rate:** **> 65%** (за счет динамической адаптации длины и тональности диалога).
*   **Target Action Rate (NPS / Конверсия):** Рост в **2.5 раза** по сравнению с классическими IVR-опросами.
*   **Cost per Contact:** Снижение на **35–45%** за счет исключения звонков в непродуктивное время.
*   **Time-to-Insight:** Сокращение времени получения аналитического отчета с 5–7 дней до **< 2 часов**.
*   **Technical Latency:** End-to-end задержка голосового конвейера (ASR → LLM → TTS) **< 800 мс** (p95).

## 4. Требования к системе

### 4.1. Функциональные требования
1. **Предиктивная сегментация:** Автоматический скоринг базы клиентов по вероятности ответа (`p_answer`) и вероятности целевого действия (`p_conversion`).
2. **Динамическая генерация скриптов:** Создание 3–5 вариаций opening lines и сценариев на лету, в зависимости от профиля клиента и результатов A/B/n тестирования.
3. **Real-time Voice Interaction:** Полнодуплексный голосовой диалог с функцией Barge-in (перебивание) и задержкой < 800 мс.
4. **Мультимодальный анализ эмоций:** Детекция тональности по тексту (NLP) и акустическим признакам (просодия: тон, темп, паузы) в реальном времени.
5. **Эпизодическая память:** Использование контекста предыдущих взаимодействий для персонализации («Иван Иванович, в прошлый раз вы упоминали проблему с доставкой...»).
6. **Автономное обучение (Reflection):** Пост-обработка диалогов для выявления причин отказов и автоматическая корректировка промптов/скриптов.
7. **CRM-синхронизация:** Автоматическое создание карточек, простановка тегов, обновление NPS-скоринга и передача «горячих» лидов живым операторам.

### 4.2. Нефункциональные требования
*   **Throughput:** Масштабируемость до 500+ одновременных голосовых сессий на одном кластере (горизонтальное масштабирование воркеров).
*   **Доступность:** 99.9% uptime для критических компонентов (оркестратор, очереди).
*   **Наблюдаемость:** Полное логирование транзакций, метрик задержки, использования токенов и решений агента в Prometheus/Grafana.

### 4.3. Требования безопасности и РФ-специфика (Compliance)
*   **152-ФЗ:** Локализация всех данных (аудиозаписи, транскрипты, PII) на серверах в РФ. Использование on-premise LLM для обработки чувствительных данных.
*   **38-ФЗ «О рекламе»:** Строгая проверка флага `consent_to_call` в CRM перед добавлением номера в очередь обзвона.
*   **Приказ Минцифры № 291:** Обязательное воспроизведение фразы «Этот разговор может быть записан в целях контроля качества» в первые 3 секунды соединения.
*   **Роскомнадзор (Frequency Capping):** Встроенный лимит: не более 2 исходящих контактов в неделю на один номер.
*   **Blacklist API:** Интеграция с внутренними и внешними реестрами «Не беспокоить» (DNC).

## 5. Архитектура системы
Система строится по принципу **Event-Driven Multi-Agent Architecture**:
1. **Data & State Layer:** PostgreSQL 16 (+pgvector), Redis 7.2 (Streams для очередей), MinIO (S3 для аудио), Qdrant (векторная БД для RAG и памяти).
2. **ML & AI Core Layer:** CatBoost (Propensity Engine), LangGraph (LLM Orchestrator), Mem0 (Memory Manager), MLflow (MLOps).
3. **Real-time Voice Layer:** Active Call (Rust) + Asterisk (SIP bridge), LiveKit Agents (WebRTC media pipeline), Faster-Whisper / Cohere Transcribe (ASR), Silero / Yandex SpeechKit (TTS).
4. **Integration Layer:** Composio (унифицированный доступ к API CRM, мессенджеров), Telephony Gateway (SIP-транки).

### Этап 2. Разработка пропенсити-модели (Weeks 5–7)
**Цель:** создание ML-модели, прогнозирующей `p_answer` и `p_conversion`.

| Действие | Детализация | Технологии |
|---|---|---|

### Этап 3. Разработка LangGraph-оркестрации (Weeks 7–9)
**Цель:** создание мультиагентного графа, управляющего кампанией.

| Действие | Детализация | Технологии |
|---|---|---|
| **Определение графа состояний** | Реализовать 6 состояний: PLANNING, HUMAN_APPROVAL, DIALING, OPTIMIZING, REFLECTING, DONE. | LangGraph, TypedDict |
| **Campaign Planner Agent** | Агент получает цель. Использует пропенсити-модель для отбора top-10 000 клиентов по `p_answer * p_conversion`. Генерирует 3–5 вариантов скриптов через LLM. | LangGraph, LLM (YandexGPT/Qwen) |
| **Human-in-the-loop блок** | Реализовать механизм паузы: граф доходит до HUMAN_APPROVAL, генерирует Task-объект в брокере, ждёт confirm/reject от супервайзера через Telegram-бота или Web UI. | Redis Streams, python-telegram-bot |
| **Dialer Orchestrator** | Вычитывает из очереди до 1000 задач, распределяет по воркерам LiveKit. Управляет rate limiting. Соблюдает frequency capping и blacklist check. | Redis Streams (consumer groups), asyncio |
| **Bandit Optimizer** | Реализовать Thompson Sampling: для каждого скрипта ведём счётчики successes/failures. После каждого звонка обновляем Beta-распределение и пересчитываем веса трафика. | numpy, scipy.stats |

**Критерий перехода:** LangGraph-пайплайн проходит сквозной тест. Bandit-алгоритм сходится к лучшему скрипту на синтетических данных за <200 итераций.

### Этап 4. Интеграция памяти Mem0 (Weeks 9–10)
**Цель:** добавление эпизодической памяти клиента для персонализации диалогов.

| Действие | Детализация | Технологии |
|---|---|---|
| **Установка и настройка Mem0** | Развернуть Mem0 на кластере (бэкенд Qdrant для on-premise). Настроить `long_term` и `session` storage. | Mem0, Qdrant, Docker |
| **Структура памяти клиента** | Определить схему: `user_id`, `conversation_summary`, `preferences`, `emotional_triggers`, `last_contact_date`, `next_best_action`. | Mem0 SDK, Python |
| **Автоматическое извлечение (auto-recall)** | При инициализации звонка Voice Agent вызывает Mem0 с `user_id`. Mem0 возвращает top-5 релевантных воспоминаний (с **Memory Decay** — недавние получают буст). Они подставляются в LLM-промпт. | Mem0 API, LangChain |
| **Обновление памяти после звонка** | Reflection Agent после звонка формирует краткое саммари, извлекает новые инсайты и вызывает `memory_add(longTerm)` для кросс-сессионного сохранения. | Mem0 API |
| **Temporal Reasoning тестирование** | Проверить корректную работу на кейсах: «в прошлом месяце клиент жаловался на доставку, а сейчас сменил адрес — не путать эти два состояния». | Mem0 evaluation suite (LoCoMo) |

**Критерий перехода:** Voice Agent успешно персонализирует диалог на основе истории (тест «слепой» vs «c памятью» — конверсия вырастает на 15–20%).

### Этап 5. Внедрение Reflection Agent (Weeks 10–11)
**Цель:** автоматическое обучение системы на результатах кампаний.

| Действие | Детализация | Технологии |
|---|---|---|
| **Сбор мета-данных кампании** | Dialer Orchestrator логирует в PostgreSQL для каждого звонка: `user_id`, `script_id`, `duration`, `barge_in_events`, `outcome`, `transcript`, `sentiment_score`. | PostgreSQL, asyncpg |
| **LLM-анализ неудач** | Для каждого отказного звонка Reflection Agent через LLM анализирует транскрипт: «Что пошло не так? Был ли вопрос слишком навязчивым? Тон агента соответствовал ожиданиям?». | YandexGPT / Llama 3.1, Structured Output (JSON) |
| **Извлечение общих инсайтов** | Агент кластеризует ошибки по сегментам: «Для сегмента 25–35 лет вопрос о зарплате вызывает негатив», «Для пенсионеров фраза «укажите, пожалуйста» воспринимается формально». | LLM + simple keyword clustering |
| **Дообучение пропенсити-модели** | Новые размеченные данные инкрементально добавляются в тренировочный датасет. CatBoost поддерживает online learning через `fit` с `init_model`. Модель переобучается каждые 24 часа. | CatBoost, MLflow |
| **Пополнение базы знаний** | Лучшие практики, выявленные в ходе кампании, добавляются в system prompt (Few-shot examples) для будущих кампаний. | LangChain, векторная БД |

**Критерий перехода:** Reflection Agent генерирует actionable инсайты (не менее 10 на кампанию). Модель улучшает ROC-AUC на 2–3% после дообучения.

## 8. Глубокий анализ и управление рисками

*   **Количественные метрики проекта:** 7 основных этапов, 18+ ключевых библиотек, интеграция с 6–7 внешними системами. Целевая задержка voice pipeline: < 800 мс. Пропускная способность: 1 LiveKit-агент ≈ 60 звонков/час.
*   **Сложность алгоритмов:** **9/10**. Комбинация временных рядов (CatBoost), генеративного планирования (LangGraph), потоковой обработки аудио и reinforcement learning (Bandit) требует высокой квалификации команды (MLOps + AI Engineers).
*   **🔍 Качественный анализ (Скрытые риски и Mitigation):**
    1.  **Ловушка "слишком умного" робота (Creepiness Factor):** Персонализация на грани ("Вы же вчера смотрели тариф на сайте") может вызвать отторжение. *Mitigation:* Настройка уровня персонализации (Consent Level) в промптах, использование только данных, явно предоставленных клиентом.
    2.  **Cost Explosion (Взрыв затрат на токены):** Генерация уникального скрипта и анализ эмоций для каждого из 10 000 звонков могут быть дорогими. *Mitigation:* Кэширование успешных скриптов для кластеров похожих клиентов; использование дешевых моделей (Qwen-7B) для рутины и YandexGPT только для сложных возражений.
    3.  **Propensity Model Bias:** Модель может научиться "игнорировать" определенные демографические группы (например, пожилых людей), если исторически они реже брали трубку. *Mitigation:* Регулярный fairness-аудит модели, добавление штрафных санкций в функцию потерь за игнорирование миноритарных групп.
    4.  **Acoustic False Positives:** Шум на линии (улица, офис) может быть интерпретирован как "агрессия". *Mitigation:* Предварительная очистка аудио через RNNoise, калибровка порогов эмоций только для сегментов с высоким SNR (Signal-to-Noise Ratio).
    5.  **Echo Chamber Effect:** Bandit-алгоритм может слишком быстро сойтись на "безопасном", но неинновационном скрипте. *Mitigation:* Внедрение параметра $\epsilon$ (epsilon-greedy) или принудительное выделение 10% трафика на случайные, "рискованные" гипотезы для исследования (Exploration vs Exploitation).
