# Требования к проекту: VoiceGraph: Autonomous Predictive Outbound Agent

## 1. Введение и область применения
**VoiceGraph: Autonomous Predictive Outbound Agent** — это фреймворк для создания полностью автономных AI-агентов исходящих кампаний (NPS, canvassing, collections). Система заменяет классические автообзвонщики, объединяя предиктивную аналитику, оркестрацию мультиагентных графов, низколатентный голосовой конвейер и эпизодическую память для гиперперсонализированных диалогов с механизмом самообучения.

**Целевая аудитория:** Крупные контакт-центры, BPO-провайдеры, телекоммуникационные компании, банки и e-commerce, работающие на рынке РФ и соблюдающие строгие нормы 152-ФЗ и 38-ФЗ.

---

## 2. Бизнес-требования и Ключевые показатели эффективности (KPIs)

### 2.1. Бизнес-цели
1. Снижение стоимости контакта (Cost per Contact) за счет исключения холостых звонков.
2. Рост конверсии в целевое действие (NPS, продажа, анкетирование) за счет персонализации.
3. Полная автоматизация пост-аналитики кампаний (сокращение Time-to-Insight с дней до часов).

### 2.2. Измеримые KPI (Критерии приемки системы)
*   **Contact Rate (Доля дозвона):** > 40% (при базовом уровне 25%).
*   **Completion Rate (Доля дослушавших до конца):** > 65%.
*   **Target Action Rate:** Рост в 2.5 раза по сравнению с классическими IVR-роботами.
*   **Technical Latency (P95):** End-to-end задержка голосового конвейера (ASR → LLM → TTS) **< 800 мс**.
*   **ML Performance:** ROC-AUC модели предиктивного скоринга **> 0.85**, log_loss < 0.35.
*   **API Performance:** Время инференса модели CatBoost < 10 мс на батч из 1000 записей.

## 3. Функциональные требования

### 3.1. Управление кампаниями и Оркестрация (LangGraph Core)
*   **FR-1.1:** Система должна поддерживать 6 состояний графа: `PLANNING`, `HUMAN_APPROVAL`, `DIALING`, `OPTIMIZING`, `REFLECTING`, `DONE`.
*   **FR-1.2:** Узел `PLANNING` должен автоматически запрашивать скоринг базы через ML-сервис и генерировать 3–5 вариантов скриптов через LLM на основе цели кампании.
*   **FR-1.3:** Узел `HUMAN_APPROVAL` должен реализовывать механизм прерывания (`interrupt`), отправлять план супервайзеру через Telegram-бот/Web UI и ожидать явного подтверждения (`approve`/`reject`) перед запуском обзвона.
*   **FR-1.4:** Узел `DIALING` должен использовать алгоритм **Thompson Sampling (Multi-Armed Bandit)** для динамического выбора скрипта, обновляя параметры Beta-распределения (`alpha`, `beta`) после каждого завершенного звонка.
*   **FR-1.5:** Система должна обеспечивать персистентность состояния графа (Checkpointing) в Redis с включенным AOF для восстановления после сбоев подов.

### 3.2. Голосовой конвейер реального времени (Real-time Voice Pipeline)
*   **FR-2.1:** Система должна принимать WebRTC-поток от LiveKit и транслировать его в SIP-транк через кастомный шлюз (Active Call/Rust) с поддержкой SIPREC.
*   **FR-2.2:** Реализация VAD (Voice Activity Detection) на базе Silero должна обеспечивать обнаружение начала речи клиента и прерывание TTS (barge-in) с задержкой **< 300 мс**.
*   **FR-2.3:** ASR (Faster-Whisper) и LLM (vLLM) должны работать в режиме строгого стриминга (chunk-by-chunk), передавая частичные результаты для минимизации TTFT (Time-To-First-Token).
*   **FR-2.4:** Параллельно с ASR, аудиопоток должен анализироваться модулем `Emotion Detector` (OpenSMILE) для классификации эмоций (`CALM`, `ANNOYED`, `CONFUSED`, `ANGRY`) и динамической адаптации системного промпта LLM.

### 3.3. Предиктивное скорирование (ML Core)
*   **FR-3.1:** ML-сервис должен принимать на вход список `user_id` и возвращать отсортированный список с рассчитанным `priority_score = p_answer * p_conversion`.
*   **FR-3.2:** Модель CatBoost должна использовать не менее 50 признаков (исторические, поведенческие, контекстные, демографические).
*   **FR-3.3:** Выходные вероятности модели должны быть откалиброваны с использованием метода Platt Scaling для обеспечения соответствия предсказанной и эмпирической вероятности.

### 3.4. Эпизодическая память и персонализация (Mem0)
*   **FR-4.1:** Перед началом звонка агент должен выполнять запрос `mem0.search` для извлечения топ-5 релевантных фактов о клиенте.
*   **FR-4.2:** Система должна применять механизм **Memory Decay**: вес факта должен экспоненциально уменьшаться в зависимости от времени, прошедшего с момента его создания.
*   **FR-4.3:** После завершения звонка `Reflection Agent` должен извлекать новые факты (предпочтения, жалобы) из транскрипта и вызывать `mem0.add` для кросс-сессионного сохранения.

### 3.5. Самообучение и аналитика (Reflection Agent)
*   **FR-5.1:** Система должна асинхронно логировать метаданные каждого звонка (`outcome`, `transcript`, `sentiment_score`) в PostgreSQL.
*   **FR-5.2:** Для звонков с исходом `REFUSAL` или `ANGRY`, LLM должна генерировать структурированный инсайт (причина отказа, предложенное изменение скрипта) в формате JSON (Pydantic Structured Output).
*   **FR-5.3:** Еженедельно система должна агрегировать инсайты и автоматически обновлять Few-Shot примеры в промптах узла `PLANNING` для будущих кампаний.

### 3.6. Интеграции и бизнес-логика
*   **FR-6.1:** Интеграция с CRM (Битрикс24 / amoCRM) через Composio должна обеспечивать создание сделок, обновление полей и создание задач в реальном времени.
*   **FR-6.2:** Реализация паттерна **Circuit Breaker** для всех внешних API: при 5 последовательных ошибках 5xx система должна временно кэшировать действия в Redis и ставить задачу на ретрай, не прерывая голосовой конвейер.
*   **FR-6.3:** По завершении кампании система должна автоматически генерировать PDF-отчет (WeasyPrint) с метриками и инсайтами и отправлять его супервайзеру.

## 4. Нефункциональные требования

### 4.1. Производительность и Масштабируемость
*   **NFR-1.1:** Голосовой конвейер (LiveKit Worker) должен поддерживать горизонтальное масштабирование (HPA в Kubernetes) на основе метрики `active_webrtc_sessions` до 500+ параллельных звонков на кластер.
*   **NFR-1.2:** vLLM должен быть настроен с `tensor_parallel_size=4` и `enable_prefix_caching=true` для обеспечения пропускной способности не менее 50 запросов в секунду на кластер H100.
*   **NFR-1.3:** Очереди задач должны обрабатываться через Redis Streams с использованием Consumer Groups для гарантированной доставки и балансировки нагрузки между воркерами.

### 4.2. Надежность и Отказоустойчивость
*   **NFR-2.1:** Доступность (Uptime) критических компонентов (LangGraph Orchestrator, Voice Workers, API Gateway) должна составлять **99.9%**.
*   **NFR-2.2:** Все состояния диалогов и кампаний должны сохраняться в Redis с политикой `appendonly yes` (AOF) для предотвращения потери данных при перезапуске подов.
*   **NFR-2.3:** Система должна поддерживать Graceful Degradation: при недоступности LLM агент должен корректно завершать звонок с сообщением о технических работах и ставить задачу оператору, а не "зависать".

### 4.3. Безопасность и Соответствие регуляторным требованиям РФ (Compliance)
*   **NFR-3.1 (152-ФЗ):** Все данные (аудиозаписи, транскрипты, PII) должны храниться и обрабатываться исключительно на серверах, физически расположенных на территории РФ. Использование внешних облачных LLM запрещено для данных с ПДн (требуется on-premise vLLM или сертифицированный контур Яндекса/Сбера).
*   **NFR-3.2 (38-ФЗ "О рекламе"):** Система должна жестко блокировать добавление в очередь обзвона любых записей, у которых флаг `consent_to_call` в CRM равен `false` или `null`.
*   **NFR-3.3 (Приказ Минцифры № 291):** В первые 3 секунды после поднятия трубки система обязана воспроизводить фразу: *"Этот разговор может быть записан в целях контроля качества"*.
*   **NFR-3.4 (Роскомнадзор):** Встроенный механизм **Frequency Capping**: система должна проверять историю и блокировать более 2 исходящих контактов на один номер в течение 7 дней (если не инициирован обратный звонок).
*   **NFR-3.5 (PII Masking):** Все текстовые данные, передаваемые в LLM или сохраняемые в логи, должны проходить через middleware `Microsoft Presidio` для замены номеров паспортов, карт, СНИЛС и телефонов на токены `[REDACTED]`.

## 5. Требования к данным и MLOps

### 5.1. Управление данными (Data Management)
*   **DM-1.1:** Все сырые и обработанные датасеты для обучения должны версионироваться с использованием **DVC** (Data Version Control).
*   **DM-1.2:** Перед обучением модели данные должны проходить валидацию через **Great Expectations** (проверка на полноту, отсутствие дубликатов, корректность типов).
*   **DM-1.3:** Аудиозаписи звонков должны храниться в S3-совместимом хранилище (MinIO) с политикой жизненного цикла (автоматическое удаление или архивация через 12 месяцев).

### 5.2. Жизненный цикл моделей (MLOps)
*   **MLO-1.1:** Все эксперименты обучения CatBoost должны логироваться в **MLflow** (параметры, метрики, артефакты модели).
*   **MLO-1.2:** Должен быть реализован конвейер инкрементального дообучения (online learning) модели CatBoost каждые 24 часа на новых размеченных данных с использованием `init_model`.
*   **MLO-1.3:** Перед деплоем новой версии модели она должна проходить автоматическое A/B тестирование или канареечный релиз (Canary Release) на 5% трафика.

### 5.3. Наблюдаемость (Observability)
*   **OBS-1.1:** Все микросервисы должны быть инструментированы **OpenTelemetry** для обеспечения сквозного трейсинга (Trace ID, привязанный к `session_id` звонка).
*   **OBS-1.2:** В Prometheus должны экспортироваться следующие бизнес- и технические метрики:
    *   `voice_latency_ms` (гистограмма)
    *   `asr_word_error_rate` (WER)
    *   `bandit_script_weights` (gauge по каждому script_id)
    *   `campaign_conversion_rate`
*   **OBS-1.3:** В Grafana должны быть настроены дашборды и алерты (в Telegram) при превышении latency > 1000 мс в течение 2 минут или при аномальном росте доли отказов.

## 6. Требования к интерфейсам (UI/UX)

Поскольку система автономна, интерфейсы требуются только для управления и контроля:
*   **UI-1.1 (Supervisor Telegram Bot):** Должен поддерживать команды `/approve_campaign [id]`, `/reject_campaign [id]`, `/get_report [id]`. Интерфейс должен отображать краткую сводку (размер аудитории, ожидаемая конверсия, ссылки на превью скриптов).
*   **UI-1.2 (Grafana Dashboards):** Должен предоставлять визуализацию в реальном времени: карта активных звонков, распределение эмоций, динамика весов Bandit-алгоритма, топ причин отказов (из Reflection Agent).

## 7. Ограничения и допущения (Constraints & Assumptions)

### 7.1. Технические ограничения
*   Для обеспечения заявленной latency (< 800 мс) требуется выделенный кластер GPU (минимум 4x NVIDIA H100 80GB или эквивалент) для инференса vLLM.
*   Качество ASR напрямую зависит от качества SIP-транка. Требуется кодек Opus или G.711 без транскодирования и с джиттером < 30 мс.
*   Алгоритм Thompson Sampling требует минимум 50-100 начальных звонков на каждый вариант скрипта для выхода на стабильную сходимость.

### 7.2. Юридические и бизнес-допущения
*   Предполагается, что заказчик (BPO или компания) уже имеет юридически корректные оферты с клиентами, содержащие согласие на обработку ПДн и использование средств автоматизации для связи.
*   Система не несет ответственности за блокировки номеров со стороны операторов связи (Spam Protection), если частотные лимиты (Frequency Capping) соблюдаются.
*   Интеграция с CRM предполагается через официальные API с использованием OAuth 2.0; прямая запись в базы данных CRM запрещена.

## 8. Критерии готовности (Definition of Done) для этапов

*   **DoD Этапа 1 (Voice):** Latency < 800 мс (p95) подтверждена нагрузочным тестом Locust на 100 параллельных сессиях. Barge-in работает корректно.
*   **DoD Этапа 2 (ML):** Модель CatBoost развернута, ROC-AUC > 0.85 на hold-out выборке, API отвечает < 10 мс.
*   **DoD Этапа 3 (LangGraph):** Сквозной тест от планирования до обзвона пройден на синтетических данных. Bandit корректно смещает веса.
*   **DoD Этапа 4 (Memory):** Агент успешно использует исторические факты из Mem0 в диалоге (подтверждено A/B тестом "слепой" vs "с памятью", рост конверсии > 15%).
*   **DoD Этапа 5 (Reflection):** Агент генерирует валидные JSON-инсайты, которые успешно обновляют промпты без вмешательства человека.
*   **DoD Этапа 6 (Final):** CRM обновляется в реальном времени, PII замаскирован во всех логах, PDF-отчет приходит автоматически, все compliance-чеклисты подписаны.
